# Feature Specification: Pipeline Output Quality Overhaul

**Feature Branch**: `009-pipeline-output-quality`
**Created**: 2026-03-27
**Status**: Draft
**Input**: User description: "Improve case analysis output quality by adding system messages, Arabic-only enforcement, RAG in Agent 0, brief post-processing, chain-of-thought reasoning, and quality gates across the 13-agent legal analysis pipeline."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Pure Arabic Legal Brief Output (Priority: P1)

As a Saudi lawyer using the system, I submit a case for analysis and expect the final legal brief to be entirely in formal legal Arabic (العربية الفصحى القانونية) — free of English terms, JSON artifacts, confidence scores, internal markers, or any technical formatting. The brief should read as if it were drafted by a professional Saudi lawyer, not generated by software.

**Why this priority**: This is the core user-facing problem. Users receive briefs with English field names (statute_id, chunk_id), CASE/LAW reference markers, confidence percentages, and mixed-language paragraphs. This makes the output unusable for court submission without extensive manual editing.

**Independent Test**: Submit a case with documents, run the full pipeline, and verify the final brief contains only Arabic legal prose — no English words (except proper nouns), no JSON blocks, no internal markers, no confidence scores.

**Acceptance Scenarios**:

1. **Given** a completed case analysis pipeline, **When** the final brief is generated, **Then** at least 95% of characters are Arabic script and no internal markers (CASE:{ref}, LAW:{ref}, confidence:, statute_id:) appear in the text
2. **Given** Agent 8 drafts the legal brief, **When** it cites a law article, **Then** the citation is written in Arabic prose form (e.g., "وفقاً للمادة الثمانين من نظام العمل") not as an internal reference marker
3. **Given** the pipeline completes all phases, **When** the user views the case output, **Then** the primary display is the final court-ready brief, not intermediate agent outputs

---

### User Story 2 - Stronger Legal Reasoning & Analysis Depth (Priority: P1)

As a legal professional, I expect the system to produce deep, case-specific legal analysis rather than generic formulaic responses. Each agent should reason through the legal issues using structured thinking — identifying key legal points, evaluating strengths and weaknesses, and building arguments anchored in specific statute articles from the RAG database.

**Why this priority**: Shallow analysis makes the system unreliable for professional use. Lawyers need detailed syllogisms (major premise → minor premise → conclusion) that reference real statutes and adapt to the specific facts of each case.

**Independent Test**: Submit a case and verify that the defense arguments contain case-specific reasoning (not template fill-in), reference actual statute articles from the RAG database, and include structured legal syllogisms.

**Acceptance Scenarios**:

1. **Given** a case is submitted for analysis, **When** Agent 8 drafts the brief, **Then** each defense argument contains a complete syllogism (legal rule → case facts → conclusion) referencing specific articles
2. **Given** the system has relevant laws in the RAG database, **When** Agents 5, 7, and 8 process the case, **Then** each agent demonstrates structured reasoning about which laws apply and why, before producing its output
3. **Given** a labor case is submitted, **When** the pipeline produces the brief, **Then** the analysis references specific labor law articles rather than generic legal principles

---

### User Story 3 - Accurate Law Identification via RAG (Priority: P2)

As a user, when I submit a case, the system should accurately identify all relevant Saudi laws and regulations by searching the legal knowledge base (RAG database) from the very beginning of the pipeline — not relying solely on LLM general knowledge to guess which laws apply.

**Why this priority**: Correct law identification in Phase 1 is the foundation for every downstream agent. If the wrong laws are identified at the start, the entire analysis chain produces irrelevant output. Currently Agent 0 does not use RAG search at all.

**Independent Test**: Submit a case involving a specific legal domain (e.g., labor dispute), verify that Agent 0 performs a RAG search and returns the correct laws from the database, and that downstream agents use those laws for analysis.

**Acceptance Scenarios**:

1. **Given** a case is submitted with intake text describing a labor dispute, **When** Agent 0 processes it, **Then** Agent 0 searches the RAG database for relevant statutes before identifying required laws
2. **Given** the RAG database contains نظام العمل (Labor Law), **When** a labor case is analyzed, **Then** the required laws list includes نظام العمل with specific relevant articles
3. **Given** Agent 0 identifies required laws via RAG search, **When** Agent 3 builds the statute index, **Then** the statute index contains accurate article content copied from the database (not LLM-generated text)

---

### User Story 4 - Quality Gate Before Publication (Priority: P2)

As a system administrator, I want the pipeline to enforce a quality gate on the final brief before marking the case as "completed." If the brief fails validation checks (missing sections, English leakage, structural issues), the case should be marked as "completed with issues" rather than silently publishing a low-quality output.

**Why this priority**: Currently the system can mark a case "completed" even when the brief contains violations. This creates false confidence in output quality.

**Independent Test**: Deliberately produce a brief with quality issues (missing preamble, English text), run the quality gate, and verify the case is marked "completed_with_issues" rather than "completed."

**Acceptance Scenarios**:

1. **Given** Agent 9 or 12 produces a final brief, **When** the quality gate runs, **Then** the brief must pass all programmatic checks (8-section structure, Arabic-only content, preamble present) to be marked "completed"
2. **Given** a brief fails the quality gate, **When** the pipeline finishes, **Then** the case status is set to "completed_with_issues" and the specific failures are logged
3. **Given** a critical agent (6, 8, or 9) exhausts its 3 self-correction attempts, **When** the pipeline evaluates whether to continue, **Then** the pipeline halts rather than passing known-bad output downstream

---

### User Story 5 - Consistent Agent Persona & Behavior (Priority: P3)

As a system developer, I want each agent to have a dedicated identity (system message) that anchors its behavior as a specific type of Saudi legal expert. This ensures consistent Arabic output style, stronger rule adherence, and role-appropriate analysis across all 13 agents.

**Why this priority**: Without system messages, each agent treats its instructions as a generic user request. Adding system messages with Arabic legal personas improves output quality, consistency, and rule adherence — especially for long-context prompts.

**Independent Test**: Run the same case twice — once with the old single-user-message approach, once with system + user message split — and compare output quality for Arabic prose consistency and instruction adherence.

**Acceptance Scenarios**:

1. **Given** any agent in the pipeline receives a case to process, **When** it makes an LLM call, **Then** the call includes both a system message (agent identity + rules) and a user message (case context + task)
2. **Given** Agent 8 (Legal Drafter) processes a case, **When** its system message defines it as a professional Saudi legal drafter, **Then** the output uses consistent formal Saudi legal Arabic without English technical terms
3. **Given** Agent 10 (Judge) processes a case, **When** its system message defines it as a Saudi judge, **Then** the critique is written in first person from the judge's perspective

---

### Edge Cases

- What happens when the RAG database has no relevant laws for a submitted case type? The system should note the gap and proceed with best available analysis, flagging the brief as limited.
- What happens when a brief passes structural validation but still contains a few English proper nouns (e.g., company names, product names)? The Arabic-only validator should allow proper nouns while flagging technical English terms.
- What happens when Agent 9 fails to produce the FINAL_BRIEF_V2 section marker? The system should fall back to constructing v2 from the v1 brief plus any fixes applied.
- What happens when context exceeds the increased budget (100K for law library)? The system should truncate with a clear Arabic note about omitted content.
- What happens when the Puter LLM provider is used instead of OpenRouter? System messages and all quality gates must work identically with both providers.
- What happens when a case involves multiple legal domains (e.g., labor + commercial)? Agent 0's RAG search should identify laws across all relevant domains.

## Requirements *(mandatory)*

### Functional Requirements

**System Message Architecture**

- **FR-001**: System MUST send every LLM call with a system message (agent identity and rules) and a user message (case data and task), rather than a single user message containing both
- **FR-002**: System MUST provide each of the 13 agents with a unique Arabic system message defining its role, expertise, and writing style as a Saudi legal professional
- **FR-003**: System MUST separate static instructions (persona, rules, output format) into the system message and dynamic content (case context, prior outputs) into the user message

**RAG Integration for Agent 0**

- **FR-004**: System MUST perform a RAG search (vector similarity search) in Agent 0 before identifying required laws, using keywords extracted from the case intake text
- **FR-005**: System MUST include RAG search results as statute candidates in Agent 0's context, so law identification is grounded in the actual law library rather than LLM general knowledge

**Arabic-Only Output Enforcement**

- **FR-006**: System MUST instruct Agent 8 (Legal Drafter) to write all citations in Arabic prose form directly (e.g., "وفقاً للمادة الثمانين من نظام العمل") rather than using internal markers (CASE:{ref}, LAW:{ref})
- **FR-007**: System MUST validate that any final brief (v2 or v3) contains at least 95% Arabic characters and zero instances of internal technical markers
- **FR-008**: System MUST provide a deterministic post-processing step that cleans any remaining English artifacts, confidence scores, agent metadata, and JSON blocks from the final brief

**Brief Post-Processing**

- **FR-009**: System MUST run a post-processor on the final brief after Agent 9 and after Agent 12 that strips remaining markers, removes confidence scores, and validates the 8-section mandatory structure
- **FR-010**: System MUST ensure the final brief begins with "بسم الله الرحمن الرحيم" as the first line
- **FR-011**: System MUST produce a single final Arabic document that merges the best available brief version (v3 preferred, v2 fallback, v1 last resort)

**Chain-of-Thought Reasoning**

- **FR-012**: System MUST include structured thinking instructions in the prompts for Agents 5 (Law Manager), 7 (Defense Strategist), and 8 (Legal Drafter) that guide the LLM to reason through key legal points, applicable statutes, strengths/weaknesses, and opponent arguments before producing output

**Self-Correction & Quality Gate**

- **FR-013**: System MUST halt the pipeline when critical agents (6, 8, 9) exhaust their 3 self-correction attempts, rather than continuing with best-effort output
- **FR-014**: System MUST provide a fallback mechanism in Agent 9 that constructs a brief_v2 from brief_v1 plus applied fixes when the standard section marker is not found in the LLM output
- **FR-015**: System MUST run a quality gate on the final brief before setting the case status to "completed" — if the gate fails, status MUST be "completed_with_issues"
- **FR-016**: Quality gate MUST check: 8-section structure present, Arabic character ratio at least 95%, no internal markers, preamble present, three-part requests present

**Context Budget**

- **FR-017**: System MUST support a law library context of at least 100,000 characters (up from 50,000) per agent
- **FR-018**: System MUST support a statute index context of at least 120,000 characters (up from 80,000) for agents that consume the statute index

**Backward Compatibility**

- **FR-019**: All changes MUST work with both OpenRouter and Puter LLM providers without degradation
- **FR-020**: All changes MUST support real-time streaming (SSE) for output display during processing
- **FR-021**: All changes MUST be backward compatible — existing cases and their stored outputs MUST remain accessible

### Key Entities

- **System Message**: The persona and rule set for each agent, sent as the system role in LLM calls. Contains agent identity, core rules, behavior instructions, anti-hallucination rules, and output format.
- **User Message**: The case-specific content sent as the user role in LLM calls. Contains intake text, documents, law library context, prior agent outputs, and the task instruction.
- **BriefPostProcessor**: A deterministic service that cleans the final brief by stripping markers, removing English artifacts, validating structure, and enforcing Arabic-only content.
- **FinalArabicBriefComposer**: A service that selects the best available brief version (v3 > v2 > v1), applies post-processing, and produces the single court-ready Arabic document.
- **Quality Gate**: A set of programmatic validation checks that must pass before a case is marked "completed."

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Final briefs contain at least 95% Arabic characters with zero instances of internal markers (CASE:{ref}, LAW:{ref}, statute_id, chunk_id, confidence scores)
- **SC-002**: Every defense argument in the final brief includes a complete legal syllogism (major premise → minor premise → conclusion) referencing specific statute articles
- **SC-003**: Agent 0 identifies relevant laws with at least 80% accuracy compared to a human legal expert's assessment, by searching the RAG database before law identification
- **SC-004**: 100% of final briefs pass the quality gate (8-section structure, preamble, three-part requests, Arabic-only) before the case is marked "completed"
- **SC-005**: Each LLM call in the pipeline uses a system message with agent-specific persona, resulting in consistent Arabic legal writing style across all agents
- **SC-006**: The pipeline processes cases with the same or better performance (within 10% of current processing time) after the quality improvements
- **SC-007**: Agent self-correction for critical agents (6, 8, 9) halts the pipeline on exhaustion rather than passing known-bad output, reducing downstream error propagation to zero for those agents

## Assumptions

- The existing LLM providers (OpenRouter and Puter) support system messages in the API request format (system role in the messages array). This is standard for all modern LLM APIs.
- The RAG database will contain at least the currently seeded 4 laws for testing. Expanding the law library (adding more Saudi laws) is a separate future effort and is not a prerequisite for this feature.
- The Arabic character ratio threshold of 95% allows for a small number of English proper nouns (company names, product names) and standard abbreviations that naturally appear in Saudi legal documents.
- The existing case status enum in the database supports or can accommodate a "completed_with_issues" status without schema changes, or the system can reuse an existing status like "completed_with_warnings."
- Chain-of-thought instructions will use structured Arabic prompts that request reasoning steps. The thinking content may or may not appear in the final output depending on the LLM model's capabilities (some models support hidden thinking blocks).
